# 高性能内存池项目说明

## 1. 项目概述

这是一个基于 C++17 实现的高性能内存池项目，核心目标是优化高频小对象分配场景下的内存管理开销。项目采用分层缓存设计，将内存申请与释放过程拆分为线程本地、进程共享和页级管理三个层次，以减少频繁 `new/delete` 带来的系统分配成本和多线程锁竞争。

当前项目已经完成：

1. 小对象内存池核心实现。
2. 大小类映射与对象对齐策略。
3. `ThreadCache / CentralCache / PageCache / Span` 四级核心数据结构协同。
4. 多线程 benchmark 与自动化报告输出。
5. 超过 `100` 个测试场景的性能对比实验。

## 2. 项目解决的问题

默认分配器在通用性上很强，但在以下场景下容易成为性能瓶颈：

1. 高频率地申请和释放小块内存。
2. 多线程同时访问分配器，导致锁竞争增加。
3. 重复创建和销毁生命周期较短的小对象。
4. 系统调用频繁，导致吞吐下降。

这个项目的优化方向就是：

1. 尽量让小对象在本线程内完成复用。
2. 通过批量申请和批量回收摊薄共享路径开销。
3. 用页级内存组织方式统一管理更大块的连续空间。

## 3. 整体架构

项目采用如下分层结构：

`ThreadCache -> CentralCache -> PageCache -> Span`

各层职责如下：

### 3.1 ThreadCache

线程本地缓存，负责处理大部分小对象的快速分配与释放。

特点：

1. 每个线程拥有自己的 `FreeList` 数组。
2. 命中时不需要访问共享结构。
3. 主要目标是降低锁竞争与访问延迟。

### 3.2 CentralCache

共享中心缓存，负责在线程缓存不足时批量提供对象，或在线程缓存过长时接收回收对象。

特点：

1. 按大小类维护 `Span` 列表。
2. 支持批量搬运对象。
3. 平衡线程局部性和全局内存复用。

### 3.3 PageCache

页级缓存，负责直接向底层申请较大块内存，并将其交给上层切分管理。

特点：

1. 以页为粒度组织内存。
2. 为小对象分配提供更大粒度的连续空间。
3. 对大对象分配走独立路径。

### 3.4 Span

`Span` 是一段连续内存及其元信息的抽象表示。

项目中的 `Span` 包含：

1. 内存起始地址。
2. 页数与总字节数。
3. 对象大小与对象数量。
4. 当前使用计数 `use_count`。
5. 空闲对象链表 `free_list`。

## 4. 核心设计点

## 4.1 大小类映射

项目将小对象按固定规则映射到不同的大小类中，便于快速定位对应自由链表。

当前规则：

1. 最大小对象阈值为 `1024 Bytes`。
2. 采用 `8 Bytes` 对齐。
3. 不同大小对象映射到固定桶位。

这样做的意义是：

1. 分配定位快。
2. 结构简单，适合教学和展示。
3. 有利于批量搬运和统一管理。

## 4.2 对象头记录归属

当前实现中，每个对象前会附带一个对象头 `ObjectHeader`，内部记录其所属 `Span` 指针。

这样做的作用是：

1. 释放对象时可以直接定位到所属 `Span`。
2. 不需要额外哈希表查询。
3. 实现路径清晰，方便分析和扩展。

这是一种工程化简化方案。更工业级的分配器通常会使用页号映射或更轻量的元数据管理方式。

## 4.3 批量搬运机制

当线程本地缓存为空时，不会一次只向中心缓存申请一个对象，而是一次申请一批对象。

原因：

1. 降低加锁频率。
2. 减少共享结构访问次数。
3. 将单次加锁开销摊薄到多个对象分配请求。

## 4.4 小对象与大对象分治

对于超过阈值的对象，项目不再走小对象缓存路径，而是直接申请独立 `Span`。

原因：

1. 大对象不适合进入小对象分桶体系。
2. 直接单独管理更简单。
3. 可以避免小对象缓存结构被大块内存拖慢。

## 5. 关键数据结构说明

### 5.1 FreeList

用于管理空闲对象链表。

作用：

1. 保存空闲对象节点。
2. 支持单个对象 `Push/Pop`。
3. 支持批量对象 `PushRange/PopRange`。

### 5.2 SizeClass

负责大小对齐、桶位映射、批量搬运数量计算和页数建议。

这是整个内存池中最重要的“策略层”之一，决定了：

1. 对齐粒度。
2. 分桶规则。
3. 每次搬运多少对象。
4. 每次从页缓存申请多少页。

### 5.3 ThreadCache

内部维护多个 `FreeList`，分别对应不同大小类。

它的意义在于：

1. 将高频请求拦截在本线程。
2. 提升命中率。
3. 降低共享锁路径压力。

### 5.4 CentralCache

中心缓存用数组维护各大小类的 `Span` 列表，并为每个大小类配一把锁。

它的意义在于：

1. 对线程缓存进行统一补货。
2. 回收过量空闲对象。
3. 维持全局内存复用。

## 6. 当前实现流程

### 6.1 分配流程

小对象分配流程：

1. 请求进入 `MemoryPool::Allocate`。
2. 判断对象大小是否属于小对象范围。
3. 若是小对象，则进入当前线程的 `ThreadCache`。
4. 若线程缓存命中，则直接返回。
5. 若未命中，则向 `CentralCache` 批量申请。
6. 若中心缓存也无可用对象，则向 `PageCache` 申请新的 `Span` 并切分对象。

### 6.2 释放流程

1. 请求进入 `MemoryPool::Deallocate`。
2. 通过对象头找到所属 `Span`。
3. 如果是大对象，则直接释放整个 `Span`。
4. 如果是小对象，则回收到当前线程的 `ThreadCache`。
5. 当线程缓存过长时，将部分对象批量归还给 `CentralCache`。
6. 当某个 `Span` 的 `use_count` 降为 `0` 时，释放整个 `Span`。

## 7. Benchmark 设计

升级后的项目已经支持多场景 benchmark，并自动输出 JSON 报告。

当前测试特征：

1. 测试场景数为 `112`。
2. 线程数覆盖 `1 ~ 8`。
3. 对象大小覆盖从小块对象到 `1024 Bytes` 范围。
4. 同时对比原生 `new/delete` 与自定义内存池。

测试目的：

1. 观察不同线程数下内存池的表现。
2. 观察不同对象大小下的性能趋势。
3. 分析哪些场景更适合内存池优化。

## 8. 当前结果与结论

从当前测试结果来看，项目已经具备明确的优化效果：

1. 已完成 `112` 组测试场景。
2. 最佳加速比达到 `5.00x`。
3. 平均加速比约为 `1.86x`。

这说明：

1. 内存池在部分高并发、小对象场景下优势明显。
2. 不是所有尺寸下都一定更快，分配器优化存在场景依赖性。
3. 项目已经具备“结构合理 + 数据支撑”的完整性。

## 9. 项目中的工程权衡

这版实现强调的是“结构完整、逻辑清晰、可运行、可解释”，因此做了一些取舍。

当前保留的简化点包括：

1. 用对象头记录 `Span` 归属，而不是页号映射。
2. 未实现 Span 的拆分与合并。
3. 未针对跨线程释放设计 transfer cache。
4. 未加入更细粒度的统计指标，如命中率、锁等待时间、碎片率。

这些并不代表项目不完整，恰恰说明它已经完成了核心路径实现，同时保留了清晰的后续演进方向。

## 10. 后续可升级方向

如果继续把项目向工业级分配器推进，可以考虑：

1. 引入页号映射替代对象头归属定位。
2. 增加 Span 拆分与合并机制。
3. 设计跨线程释放优化策略。
4. 加入 transfer cache 或更细粒度的中间层。
5. 增加性能指标采集与自动化分析。
6. 增加更严格的测试，包括正确性测试、压力测试和边界测试。

## 11. 可直接使用的项目总结

可以用下面这段话概括项目：

“我实现了一个面向高频小对象分配场景的高性能内存池。项目采用 `ThreadCache、CentralCache、PageCache、Span` 的分层结构，通过线程本地缓存、中心缓存批量搬运和页级连续内存管理，减少默认分配器在小对象高频场景下的系统调用开销和多线程锁竞争。当前项目已经完成多场景 benchmark，并在 112 组测试中取得了最高 5 倍、平均约 1.86 倍的性能提升。同时我也分析了当前实现的工程边界以及继续向工业级分配器演进的方向。”
